# QBastion Architecture

## Overview

QBastion is a quantum-resistant node protection layer for cryptocurrency networks. Its primary goal is to detect and reject transactions signed with quantum-vulnerable keys, protecting wallets against future cryptanalytic attacks by sufficiently powerful quantum computers.

The system is built around two core properties:

1. **Verifiability** — Signature verification is done inside a zkVM (Cairo + Stwo), so the verification proof itself is compact and can be propagated through the network without re-executing the full verification.
2. **Quantum resistance** — The signature schemes used (Falcon, Sphincs+) are NIST post-quantum finalists, providing security against both classical and quantum adversaries.

## Signature Schemes

### Falcon-512 (Primary)

- **Type:** Lattice-based (NTRU lattice, GPV framework)
- **Security level:** NIST Level I (≈ 128-bit classical, ≈ 103-bit quantum)
- **Key size:** ~897 bytes (public key), ~1281 bytes (private key)
- **Signature size:** ~666 bytes
- **Core operation:** Discrete Gaussian sampling over NTRU lattice
- **Cairo implementation:** `packages/falcon/` — uses NTT-based polynomial multiplication

### Sphincs+-128s (Fallback)

- **Type:** Hash-based (stateless)
- **Security level:** NIST Level I
- **Signature size:** ~7856 bytes
- **Variants supported:**
  - SHA-2 simple mode
  - BLAKE2s with 4-byte aligned address encoding
- **Cairo implementation:** `packages/sphincs/` (planned)

## Cairo Verification Pipeline

### Polynomial Arithmetic

The Falcon verifier in Cairo relies on efficient polynomial multiplication in
Z_q[x] / (x^n + 1) with q = 12289. This is implemented as:

1. **NTT** (`packages/falcon/src/ntt_constants.cairo`) — precomputed roots of unity for all supported ring degrees
2. **STARK-friendly BoundedInt arithmetic** — generated by `hydra/bounded_int_circuit/`
3. **Falcon verifier entry-point** (`packages/falcon/src/lib.cairo`)

### Ingress Gateway

`packages/falcon/src/ingress.cairo` implements the `AuthorizationRequest` struct
and dispatches incoming signature bundles to the correct verifier. It validates:

- Header byte (encodes Falcon degree)
- Salt (40 bytes for Gaussian randomness)
- Encoded signature s1
- Public key hash consistency

## Python Reference Layer (`hydra/`)

The `hydra/falcon_py/` module serves two purposes:

| Purpose | Files |
|---------|-------|
| Signing (test key generation) | `falcon.py`, `ntrugen.py`, `ffsampling.py` |
| KAT vector generation | `scripts/` |
| Cairo code generation | `hydra/bounded_int_circuit/`, `hydra/compilable_circuits/` |

It is **not** hardened for production — use the Cairo verifier on-chain and the Python layer only for development and testing.

## Security Model

| Threat | Mitigation |
|--------|------------|
| Quantum key-recovery on ECDSA | Falcon / Sphincs+ PQ signatures required |
| Forged PQ signatures | Cairo verifier + STARK proof guarantees soundness |
| Signature replay | Salt binding + commitment in ingress gateway |
| Verifier bugs | STARK soundness — prover must witness a valid execution trace |
